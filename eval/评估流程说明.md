# 评估流程说明（metrics.py 与 eval harness）

本文档解释 `eval/` 下两个核心文件 `metrics.py` 与 `run_eval.py` 的内部逻辑，以及整个离线评估流程的运作方式。使用方法请见同目录下的 `Readme.md`。

## `eval/metrics.py` —— 评分与报告引擎

严格来说，它**不是 benchmark**（benchmark 是 `dataset/labeled_emails.jsonl` 那份带标签的数据集），而是**评分 / 报告引擎**。它负责把每个样本的原始结果聚合成指标，并渲染成可读报告。这个文件里完全没有调用 LLM、也不知道邮件是什么东西——它只是纯粹的数据变换层。

### 数据结构（`metrics.py:11-57`）

- `CaseResult` —— 单个邮件的评估结果：`case_id, expected, predicted, latency_ms, error`。`correct` 的判定：没有 error **且** expected == predicted。
- `CategoryStats` —— 单个类别的 `precision / recall / f1 / support`。
- `EvalReport` —— 汇总报告：总数、准确率、各类别指标、混淆矩阵、延迟分位数、以及原始每条样本。

### 指标计算

- `_per_category_stats`（`:68-98`）—— 按 `("action", "fyi", "spam")` 三类统计 TP / FP / FN，推导 precision、recall、F1。**调用出错的样本会记为其期望类别的 FN**（即召回被扣分），保证稳定性问题不会被无视。
- `_confusion`（`:101-107`）—— 生成 3×3 的 `expected → predicted` 混淆矩阵（跳过出错的样本）。
- `_percentile`（`:60-65`）—— nearest-rank 方法计算 p95；`build_report` 用 `statistics.median` 计算 p50，另外算一个 mean。
- `build_report`（`:110-127`）—— 评估流程唯一对外的入口函数，把 `list[CaseResult]` 折叠成一个 `EvalReport`。

### 报告渲染

- `render_markdown`（`:130-163`）—— 生成 `.md` 报告：顶部摘要、各类别指标表、混淆矩阵、以及**误分类明细表**（逐条列出所有判错或出错的样本，方便人工看 bad case）。

## 整体评估流程（`run_eval.py`）

1. **解析 CLI 参数**（`:62-67`）—— `--dataset`、`--results-dir`、`--limit`、`--tag`。默认指向 `eval/dataset/labeled_emails.jsonl` 和 `eval/results/`。
2. **环境预检**（`:69-75`）—— 没有 `GOOGLE_API_KEY` 直接拒绝；检查数据集存在。
3. **加载被测 agent**（`:77`）—— 从 `backend/app/services/agent_core` 导入 `categorize_logic` 和 `llm`。**注意这就是 LangGraph 节点在生产环境真实使用的那个函数**，没有 mock，保证评估口径与生产一致。
4. **加载数据集**（`:21-29, 79-81`）—— JSONL 格式，每行一个带标签的邮件（`id, sender, subject, email_content, expected_category, notes`）。
5. **逐条执行**（`run_case`，`:32-58`）—— 构造节点期望的 state dict，用 `time.perf_counter()` 计时，从返回结果中取 `result["category"]` 作为预测。**异常会被捕获并记入 `error`**，这样单条挂掉不会中断整轮评估。
6. **实时进度**（`:85-89`）—— 每条样本打印 ✓ / ✗ / ERR 一行，包含 expected、predicted 与延迟。
7. **聚合**（`:91`）—— 调用 `build_report(results)`，由 `metrics.py` 完成全部数学计算。
8. **持久化**（`:92-99`）—— 往 `eval/results/` 写入两个带时间戳的文件：
   - `eval-YYYYMMDD-HHMMSS[-tag].json` —— 完整机器可读报告（`EvalReport.to_dict()`）
   - `eval-YYYYMMDD-HHMMSS[-tag].md` —— 渲染后的 markdown
9. **打印总结并退出**（`:101-110`）—— 输出准确率 / 延迟 / 各类别 F1；**没有错误返回 0，有错误返回 1**，这样可以放到 CI 里作为门禁。

## 心智模型

```
labeled_emails.jsonl ──► run_case(categorize_logic) ──► [CaseResult, ...] ──► build_report ──► EvalReport ──► JSON + Markdown
                                  (被测 agent)                 (metrics.py)          (metrics.py)
```

两层职责分得很清楚：

- **`metrics.py`** 负责**算数 + 排版**：给我一堆 `CaseResult`，我还你一份报告。
- **`run_eval.py`** 是**驱动层**：把标注数据喂给真正的 agent，再把结果交给 `metrics.py`。

由此带来两种替换方式：

- 换 **dataset**（数据集）——改变的是「测什么」
- 换 **`categorize_logic`**（被测对象）——改变的是「测谁」
